## 1. Architecture summary

The Reliability Gateway routes requests through multiple layers of resilience:

```
User Request
    |
    v
[Gateway] ---> [Cache check (Memory/Redis)] ---> HIT? return cached (0ms latency, $0 cost)
    |                                                |
    v                                                v MISS
[Circuit Breaker: Primary] -------> Provider A (Primary)
    |  (OPEN? skip A)
    v
[Circuit Breaker: Backup] --------> Provider B (Backup)
    |  (OPEN? skip B)
    v
[Static fallback message] ("The service is temporarily degraded. Please try again soon.")
```

## 6. Redis shared cache

### Why in-memory cache is insufficient for multi-instance deployments
In-memory cache is isolated inside a single application process or container instance. In a multi-instance production environment behind a load balancer:
*   Each instance suffers from a separate **cold start** cache penalty.
*   Duplicate expensive requests for the same query are sent to the LLM providers from different instances.
*   Cache states can become inconsistent if keys are evicted or updated asynchronously.

### How `SharedRedisCache` solves this
`SharedRedisCache` solves this by storing all cache records in a centralized Redis database.
*   **Single Source of Truth:** All gateway instances read and write from/to the same shared database.
*   **High Hit Rate:** Once a query is cached by any instance, all other instances immediately benefit from it.
*   **TTL Management:** Centralized Redis handles TTL expiration automatically.

### Evidence of shared state
The test suite validates that multiple independent cache instances can query the same data successfully:
```python
c1 = SharedRedisCache(redis_url="redis://localhost:6379/0", ttl_seconds=60, similarity_threshold=0.5, prefix="rl:test:shared:")
c2 = SharedRedisCache(redis_url="redis://localhost:6379/0", ttl_seconds=60, similarity_threshold=0.5, prefix="rl:test:shared:")
c1.flush()
c1.set("shared query", "shared response")
cached, _ = c2.get("shared query")
assert cached == "shared response" # Assert passes successfully!
```

## 8. Failure analysis

### What could still go wrong?
Currently, the **circuit breaker state** (e.g. failure count, open/closed state) is stored entirely **in-memory** in the local class instances.
In a multi-instance production environment:
1.  **Split-brain Breakers:** One instance may trigger a circuit open due to local networking, while other instances keep flooding a struggling provider, causing localized request failures.
2.  **Uneven Failures:** If instance A experiences 2 failures and instance B experiences 2 failures (with a threshold of 3), the circuit remains closed on both, missing the aggregate system failure.

### What would you change?
I would store the **circuit breaker state in Redis** (using atomic counters and status flags with locks/TTL). This ensures that if any instance triggers a breaker, the state is immediately propagated globally, allowing the entire gateway cluster to fail fast.

## 9. Next steps

1.  **Redis-backed Circuit Breaker State:** Share breaker statuses globally across instances to prevent localized failures.
2.  **Cost-aware Routing:** Add budget tracking inside `complete()` to drop expensive providers and route only to cache or cheaper fallback providers when the budget threshold (e.g. 80%) is reached.
3.  **Sliding Window for Circuit Breaker:** Implement a sliding time window (rather than consecutive counters) to handle sporadic errors more gracefully.
